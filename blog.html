Algorithm Deep Dive

How to search 12 million paths in 40ms

← Back to Solver

Introduction

Wordament is a 4x4 grid word-search game where you connect adjacent letters to form words. While it looks simple, solving a board programmatically exposes a classic computer science problem: state space search space explosion.

A naive algorithm can take several seconds to solve a single board. By using proper algorithms and data structures, we can solve it in 40 milliseconds, even inside a web browser. In this post, we'll analyze different search strategies and data structures, aimed at computer science students looking to understand how algorithmic efficiency beats brute-force computing.

The Problem Space: Exponential Explosion

A 4x4 board has 16 tiles. To find a word, you can start on any of the 16 tiles, and move to any of the 8 surrounding neighbors (up, down, left, right, and diagonals), without reusing the same tile twice in the same word.

This is modeled as a directed acyclic graph (DAG) traversal for each word. If we try to enumerate every single path on the board from length 1 to 16, the branching factor is initially 8, then 7, then 6...

Summing all paths across the grid yields over 12 million unique paths. If we evaluate every path, our time complexity is exponential:

Time Complexity: O(8d) where d is the path depth.

Approach 1: Brute-Force DFS + Hash Map

The simplest approach is a brute-force Depth-First Search (DFS) to find all paths, construct the string for each path, and check if it exists in a dictionary.

To speed up lookups, you store the dictionary in a Hash Map (or HashSet), which gives O(1) average-case word validation.

Why it fails: Even though lookups are O(1), you still have to generate all 12 million paths. For every path, you allocate string memory, hash the string, and search the map, only to find it is not a word. This brute-force approach takes ~5 seconds in compiled languages like D, and would freeze the browser in JavaScript.

The Optimizer: Trie (Prefix Tree)

To solve this, we need a data structure that knows not just if a string is a word, but if a string is a valid prefix of any word. That is a Trie.

A Trie stores words as paths in a tree. Characters are nodes. Words that share prefixes share the same nodes.

Figure 1: Trie tree containing words "help", "hello", and "helper". This highlights that a valid word endpoint (* like "p") does not have to be a leaf node (it can have children like "e" and "r" extending to "helper").

With a Trie, lookup complexity is:

Lookup Complexity: O(L) where L is the length of the string, completely independent of the dictionary size N.

Approach 2: Trie-Pruned DFS (Backtracking)

Instead of doing DFS first and lookups later, we **do them simultaneously**. As we walk the grid to a new letter, we query the Trie: "Does any word in our dictionary start with this prefix?"

If the answer is **no**, we stop immediately and **backtrack**.

For example, if we start at G and move to C, the Trie says isPrefix("gc") -> False. We abort right there, saving ourselves from exploring the remaining $8^{14}$ paths starting with GC.

Figure 2: Visualizing word solving and search pruning on the same board. Grid A (left) shows the overlapping DFS paths for "help" and "hello". Grid B (right) shows how DFS immediately halts and backtracks at "c" because prefix "gc" does not exist in our Trie.

By doing this, the solver searches only ~3,000 prefix paths instead of 12 million. The state space collapses, which is why the solver takes only 40 milliseconds.

Exploring Alternatives

Let's compare this with other standard search algorithms and data structure alternatives.

1. BFS (Breadth-First Search)

Instead of DFS (which goes deep down a path first), we could use Breadth-First Search (BFS) using a Queue to explore all words of length 1, then length 2, then length 3, etc.

Space Complexity Issue: While BFS finds shorter words first, it has terrible space complexity. DFS only needs to store the current path on the call stack, which takes O(d) memory (where d ≤ 16). BFS has to store every active path of length k in its queue simultaneously. At length 10, the queue would contain hundreds of thousands of paths, consuming megabytes of memory. Thus, DFS is highly memory efficient compared to BFS.

2. Binary Search on a Sorted List

Instead of a Trie, we could store our 260k dictionary words as a flat, sorted array of strings.

To check if a string S is a valid prefix during our DFS, we can perform a Binary Search (like std::lower_bound in C++ or bisect_left in Python) to find the first word in the sorted array that is lexicographically greater than or equal to S.

Prefix matching via Binary Search: If the word found at the binary search index starts with S, then S is a valid prefix, and the search continues. If it does not start with S (or if the index is out of bounds), then no word in the dictionary starts with S. At this point, we know we have reached the limit of that search branch, and we can immediately prune and backtrack.
Trie vs. Binary Search:
  • Time Complexity: Binary search on an array of size N takes O(L · log N) per lookup. A Trie takes O(L). With N = 260,000, log2(N) ≈ 18. Theoretically, Trie lookup is faster because it does not require comparisons across a large array.
  • Space Complexity: The sorted list uses less memory (flat array of strings). The Trie requires nodes (objects) which have high memory overhead in JavaScript (~30MB for a 260k dictionary).
  • Initialization Overhead: Building the Trie requires instantiating objects for each character node. For 260,000 words, this creates millions of objects in memory, taking ~150ms of startup time in JS. Loading a pre-sorted list is a single string split into a flat array, which takes ~10ms (almost instantaneous). If the list is not pre-sorted, sorting it would require an additional O(N · log N) step.
Real-world VM Performance Note: When running both algorithms inside a web browser, you will notice that Binary Search DFS can solve the board faster than Trie DFS (often ~15ms vs ~25ms)!

This seems counter-intuitive based on Big-O analysis. However, it is caused by JavaScript engine internals (V8):
  • Trie Overhead: Walking nested JavaScript objects (node = node[char]) requires dynamic property lookups in V8. Since the Trie is built dynamically, these nodes do not share static "hidden classes" (shapes), forcing the V8 compiler to use slower "dictionary-mode" lookups.
  • Binary Search Speed: Binary search performs string comparisons on a flat array. These comparisons (like startsWith()) are executed in compiled C++ code inside the V8 engine, which is highly optimized and has a much lower constant-factor overhead.
This demonstrates how compiler architecture and VM environments can sometimes flip theoretical complexity bounds in practice.

Future Work: Trie Optimizations

While our JavaScript Trie runs in 40ms, there are advanced optimizations that could make it even more memory-efficient in a production environment:

  • Radix Tree (Compressed Trie): Merging single-child nodes (like compressing h -> e -> l -> p into a single "help" node) reduces tree depth and memory footprint.
  • Flat TypedArray Trie: Storing the Trie nodes in a single flat Int32Array (using a First-Child Next-Sibling pointer layout) avoids JavaScript object overhead, reducing memory usage from 30MB to under 3MB and eliminating Garbage Collection during initialization.

However, for a client-side board solver of this scale, the current performance is already well below the threshold of human perception. Keeping the Trie structure simple and readable is far more valuable for educational purposes than micro-optimizing memory layouts.

Conclusion

The Wordament solver shows that algorithmic design is the key to solving complex search spaces. By replacing a naive full tree search with a prefix-pruned backtracking search, we collapsed a search space of 12 million paths down to just a few thousand, resulting in a 125x speedup and making instant in-browser solving possible.
